# Autocheck — README y documentación

## Preguntas

1. **¿Qué preguntas debe responder un buen README?**
   Lista al menos tres.

2. **¿Cuál es la diferencia entre `#` y `##` en Markdown?**
   ¿Cuántos `#` de nivel 1 debería haber en un archivo?

3. **¿Por qué es problemático un README desactualizado?**
   ¿Qué efecto puede tener en alguien que llega al proyecto por primera vez?

4. **¿Qué archivos de documentación reconoce GitHub automáticamente?**
   Nombra al menos tres además del README.

5. **Sin mirar la referencia, escribe de memoria cómo se crea una tabla en Markdown.**

---

## La prueba del extraño

Pide a alguien que no conoce tu repositorio (o imagina que esa persona eres tú dentro de seis meses) que lea solo el README.

- ¿Entiende para qué sirve el proyecto?
- ¿Sabe cómo empezar a usarlo?
- ¿Tiene toda la información que necesita?

Si la respuesta a alguna es "no", el README necesita mejorar.

---

## Señales de que lo tienes controlado

- Puedes escribir un README completo en Markdown sin ayuda externa
- Tu README responde las preguntas básicas sobre el proyecto
- Usas correctamente títulos, listas, código y enlaces
- Entiendes que la documentación es parte del proyecto, no un extra
