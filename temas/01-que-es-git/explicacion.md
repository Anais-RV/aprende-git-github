# ¿Qué es Git?

## El problema que resuelve

¿Te ha pasado esto alguna vez?

```
informe.docx
informe_v2.docx
informe_v2_final.docx
informe_v2_final_DEFINITIVO.docx
informe_v2_final_DEFINITIVO_conRevisiones.docx
```

Ese caos de archivos es lo que Git viene a eliminar. Git es un **sistema de control de versiones**: una herramienta que guarda el historial completo de cambios de tus archivos, de forma ordenada y trazable.

---

## ¿Qué hace Git exactamente?

- Guarda **instantáneas** de tu proyecto en el tiempo (llamadas _commits_)
- Te permite **volver a cualquier punto anterior** del historial
- Facilita **trabajar en paralelo** sin sobrescribir el trabajo de otras personas
- Registra **quién cambió qué y cuándo**

---

## Git no es GitHub

Es importante distinguirlos desde el principio:

| | Git | GitHub |
|---|---|---|
| ¿Qué es? | Herramienta de control de versiones | Plataforma web para alojar repositorios Git |
| ¿Dónde vive? | En tu máquina | En internet |
| ¿Es obligatorio el otro? | No | No (pero se complementan bien) |

Git funciona completamente sin internet. GitHub es uno de los muchos servicios que usa Git por debajo.

---

## La metáfora del historial de guardados

Imagina un videojuego donde puedes guardar la partida en cualquier momento y darle un nombre descriptivo a cada guardado. Puedes volver a cualquiera de ellos cuando quieras. Eso es Git aplicado a tus archivos de código o texto.

La diferencia clave: en Git los "guardados" también sirven para **colaborar**. Puedes compartir tu historial con otras personas y combinar sus cambios con los tuyos.

---

## ¿Cuándo usarlo?

- En cualquier proyecto donde los archivos cambien a lo largo del tiempo
- Solo o en equipo
- Código, documentación, datos, configuraciones...

Aprenderás mucho más en los próximos temas. Por ahora lo importante es entender **el problema que resuelve**.
