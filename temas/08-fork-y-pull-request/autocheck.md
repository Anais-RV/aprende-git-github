# Autocheck — Fork y Pull Request

## Preguntas

1. **¿Qué diferencia hay entre hacer fork y hacer clone?**
   ¿Cuándo usarías uno u otro?

2. **¿Por qué no haces push directamente al repositorio original?**
   ¿Qué ocurriría si tuvieras permisos y lo hicieras?

3. **¿Qué es un Pull Request?**
   ¿Por qué se llama "pull" si eres tú quien está proponiendo los cambios?

4. **¿Cómo mantienes tu fork sincronizado cuando el proyecto original avanza?**
   ¿Qué es el remoto "upstream"?

5. **¿Por qué es buena práctica abrir un PR incluso dentro de tu propio equipo?**
   ¿Qué aporta la revisión de código?

6. **¿Qué significa que una rama local y una remota han divergido?**
   ¿Qué suele haber pasado para llegar a esa situación?

7. **Si Git rechaza un `push` con `non-fast-forward`, qué harías primero?**
   Explica el razonamiento, no solo el comando.

---

## Comprobación práctica

- ¿Tienes un PR abierto en el repositorio de práctica?
- ¿La descripción del PR explica qué hiciste y por qué?
- ¿Tu rama tiene un nombre descriptivo?

---

## Señales de que lo tienes controlado

- Puedes completar el flujo fork → clonar → rama → commit → push → PR sin ayuda
- Entiendes que un PR es una propuesta, no un comando de Git
- Sabes cómo mantener tu fork actualizado con el original
- Usas ramas descriptivas para cada contribución
- Sabes reconocer y desbloquear una situación de ramas divergentes
